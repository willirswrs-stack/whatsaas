## 1. Correção de Instâncias (Evolution API)
Identificamos que a Evolution API estava travada devido a instâncias "zumbis" antigas e dados inconsistentes no banco.
- **Ação**: Criamos e executamos um script de limpeza (`clean-evolution.js`) para remover forçosamente todas as instâncias da API.
- **Ação**: Executamos limpeza direta no banco de dados (`DELETE FROM instances`) para garantir consistência.
- **Resultado**: Ambiente da Evolution API zerado e pronto para novas conexões limpas.

## 2. Otimização de Infraestrutura (Docker)
Para preparar o projeto para produção em VPS e alta escala, removemos serviços redundantes que consumiam muita memória.
- **Removido**: Container `waha` (WAHA - Provider 1).
- **Removido**: Container `wwebjs` (WWebJS - Provider 3).
- **Mantido e Otimizado**: `evolution` (Evolution API v2).
- **Justificativa**: A Evolution API usa protocolo Baileys (sem navegador), consumindo cerca de 10x menos RAM que as outras opções, sendo a única viável para escalar centenas de chips.

## 3. Planejamento de Capacidade (Scalability)
Realizamos o dimensionamento para deploy em VPS.

| Perfil | Specs Recomendadas | Capacidade (Chips) |
| :--- | :--- | :--- |
| **Inicial** | 4 vCPU / 8 GB RAM | ~80-100 chips |
| **Intermediário** | 8 vCPU / 16 GB RAM | ~200-300 chips |
| **Alta Escala** | 16 vCPU / 32 GB RAM | ~500-800 chips |
